1. Úvod a Problém

Současný stav blockchain herního průmyslu: Většina her je buď plně on-chain (pomale, drahé) nebo jen s NFT skiny (bez herní hodnoty). Chybí skutečná integrace blockchainu do herní mechaniky.

Proč tahový systém: Tahové hry mají přirozenou synchronizaci – všechny akce se odehrávají v diskrétních krocích. Ideální pro blockchain validaci bez lagů.

Proč D20: Dvě 20-stěnné kostky dávají rozptyl 2–40 s normální distribucí. Modifikátory z vybavení (NFT) mají smysl – rozdíl mezi +2 a +5 je významný, ale ne game-breaking.

Core Hypothesis: Hráči chtějí own své itemy, obchodovat s nimi a vidět prokazatelné RNG. Blockchain to umožňuje bez důvěry v server.

2.1 Client Layer

Unity WebGL Build: 3D dungeon rendering, post-processing, particle efekty. WebGL umožňuje cross-platform bez instalace.

React UI Overlay: Inventář, tržiště, wallet panel. React pro rychlý vývoj UI, snadná integrace Web3 knihoven.

Wallet SDK: wagmi + viem pro Ethereum interakce. Podpora EIP-1193 (MetaMask), Coinbase Wallet SDK, WalletConnect.

2.2 Server Layer

Node.js + Socket.io: Real-time WebSocket komunikace. Room-based architektura pro dungeon instance.

Combat Engine: Deterministický resolver tahů. Vstup: všech 4 hráčů akce. Výstup: resolved state + loot drops.

Redis: Session storage, matchmaking queue, rate limiting. Pub/sub pro cross-server komunikaci.

PostgreSQL: Persistent data – hráčské profily, historie dungeonů, leaderboardy.

2.3 Blockchain Bridge

Serverová vrstva komunikuje s blockchainem přes:

• Read: Direct RPC calls (Alchemy/Infura) pro ziskání stavu itemů, balance.

• Write: Relayer pattern – server podepisuje transakce session key, paymaster platí gas.

3.1 Proč Multi-Chain?

Žádný jeden chain nevyhovuje všem potřebám. Trilemma: decentralizace, rychlost, cena.

3.2 Arbitrum Orbit L3 – Gaming Chain

Dedicated Layer 3 běžící na Arbitrum One. Vlastní konfigurace pro hry:

• Block time: 250ms (4× rychlejší než L2)

• Gas token: $DNG (ne ETH, native herní ekonomika)

• Precompiles: Nativní D20 roll, loot generation, combat verify

• Data Availability: Celestia/Avail pro ultra-low costs (~90% úspora)

3.3 Cross-Chain Messaging

LayerZero V2: Omnichain NFT (ONFT) a Fungible Token (OFT). Trustless bridge mezi všemi chainy.

Axelar: Fallback pro komplexnější cross-chain calls (governance, treasury).

4. Smart Contracty

4.1 ItemUpgrade.sol

Vylepšování itemů s exponenciálním risem. Chainlink VRF pro fair RNG.

// Upgrade success chance function getSuccessChance(<uint256 level) public pure returns (uint256) { return 90 - (level * 8); // 90% → 10% } function upgradeCost(<uint256 level) public pure returns (uint256) { return 0.01 ether * (2 ** level); // 0.01 → 5.12 ETH }

Fail mechanic: Při neúspěchu item downgrade o 1 level nebo destruction (dle rarity). Vytváří risk/reward tension.

4.2 ProgressNFT.sol (Soulbound)

Non-transferable achievementy. ERC-721 s override _beforeTokenTransfer.

function _beforeTokenTransfer( address from, address to, uint256 tokenId ) internal override { require( from == address(0) || to == address(0), "Soulbound: non-transferable" ); super._beforeTokenTransfer(from, to, tokenId); }

4.3 GamePaymaster.sol (ERC-4337)

Account Abstraction paymaster. Hráč neplatí gas, platí se v $DNG nebo je sponsored.

enum PayMode { FREE, DNG, SPONSORED } function _validatePaymasterUserOp( UserOperation calldata userOp, uint256 maxCost ) internal view returns (bytes memory context) { address player = userOp.sender; PayMode mode = playerPayMode[player]; if (mode == PayMode.FREE) { require(freeActions[player] > 0, "No free actions"); } else if (mode == PayMode.DNG) { uint256 dngCost = maxCost * dngPerGasUnit; require(dngToken.balanceOf(player) >= dngCost); } return abi.encode(player, mode); }

5. D20 Combat Systém

5.1 Matematika Hodu

Dvě nezávislé uniformní proměnné X, Y ~ U[1,20]. Součet S = X + Y.

E[S] = E[X] + E[Y] = 10.5 + 10.5 = 21
Střední hodnota součtu dvou D20
Var(S) = Var(X) + Var(Y) = 33.25 + 33.25 = 66.5
Rozptyl (σ ≈ 8.15)
P(S = k) = (20 - |k - 21|) / 400, pro k ∈ [2, 40]
Trojúhelníková distribuce (discrete convolution)

5.2 Critical Hit & Fumble

P(Crit) = P(X=20 ∩ Y=20) = 1/400 = 0.25%

P(Fumble) = P(X=1 ∩ Y=1) = 1/400 = 0.25%

Extrémně vzácné, ale významné. Vytváří "watercooler moments" – hráči budou vyprávět o svém critu.

5.3 Attack Resolution

Attack Roll = S + MOD_attacker ≥ AC_target
MOD = STR/DEX/INT bonus, AC = Armor Class
Damage = (Weapon_Base + MOD + d8) × Crit_Multiplier
Crit = 2×, Fumble = 0.5× self-damage

5.4 Initiative System

Initiative = D20_roll + DEX_modifier. Determinuje pořadí tahů v celém comb
